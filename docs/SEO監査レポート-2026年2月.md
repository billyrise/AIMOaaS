# AIMOaaS サイト SEO 監査レポート（2026年2月）

**目的**: 検索エンジン（Google 他）の評価基準および「AI 大量生成・低価値サイト排除」の観点に基づき、本サイトの SEO を厳しく評価し、問題を洗い出して具体策を構築する。

**参照基準**: Google Search Central 方針、Helpful Content 系アップデート、2025年1月 Quality Rater Guidelines 更新（生成 AI 評価の明文化）、Scaled Content Abuse 等のスパム定義。評価はファクトベースで記載する。

## 2. 検索エンジン・AI 低価値排除の基準（ファクト整理）

- **Google（2025年1月 QRG）**: 生成 AI を「コンテンツ作成のツール」としつつ、**メインコンテンツが自動/生成 AI で作られた低労力・スケール目的のコンテンツ**は最低品質と評価されうる。
- **Scaled Content Abuse**: 低価値コンテンツの大量生成（手動・AI 問わず）で検索結果を埋め尽くす行為はスパムとして扱う。
- **Helpful Content**: ユーザー第一・専門性・一次経験（E-E-A-T）を重視。日付・著者・独自性は「人的関与」と価値の証拠になる。

したがって、**noindex で段階的に解放する設計**と、**allowlist のみインデックス**とする現状は、リスクを抑えつつ SEO を試すうえで合理的。そのうえで、**インデックス対象ページの「価値の見え方」を高める**対策が有効。

---

## 3. 対策の全体方針

1. **インデックス対象（allowlist）を「価値が高い」と判断できるページに限定し続ける。**
2. **構造化データ・メタ・表示要素で E-E-A-T と「人的関与」を強く示す。**
3. **サイト構造で PSEO を「リソースの一部」として明示し、トップから適度にリンクする。**
4. **コンテンツの類似度・FAQ 数・内部リンク多様性をモニタし、allowlist 拡大時は品質基準を満たすページから追加する。**

以下、具体策を「実装レベル」で記載する。

---

## 4. 具体策（実装レベル）

### 4.1 Article スキーマの拡張（datePublished / dateModified / author / publisher）【実装済み】

**目的**: E-E-A-T と「いつ・誰が」を伝え、テンプレ臭を減らす。

**現状**: `scripts/pseo/gemini.ts` のプロンプトで Article は `headline`, `description` のみ。出力 HTML の JSON-LD も同様。

**実装内容**: `scripts/pseo/generate.ts` で Gemini 出力の Article に以下をマージして出力するようにした。
- `datePublished` / `dateModified`: ビルド日（`new Date().toISOString().slice(0, 10)`）
- `author`: `{ "@type": "Organization", "name": "RISEby inc.", "url": "https://riseby.net" }`
- `publisher`: `{ "@id": "https://aimoaas.com/#organization" }`
既存の PSEO HTML に反映するには `npm run generate`（または `pseo:generate`）の再実行が必要。

**具体案（今後の拡張）**:

1. **日付**
   - ビルド時またはデプロイ時に「初回公開日」「最終更新日」を 1 ページ 1 レコードで持つ（例: `data/pseo/page_dates.json`: `{ "evidence-pack-ai-audit-evidence-pack": { "datePublished": "2025-01-15", "dateModified": "2026-02-11" } }`）。
   - 未設定の場合は `dateModified` をビルド日、`datePublished` を `dateModified` と同一または過去の固定日にする。
   - `generate.ts` で Article に `datePublished` / `dateModified`（ISO 8601）を付与。

2. **著者・出版者**
   - Article に `author`: `{ "@type": "Organization", "name": "RISEby inc.", "url": "https://riseby.net" }` を固定で付与（個人名が無いため Organization で統一）。
   - `publisher`: 既存の `#organization` を参照する形で同一。

**ファイル例（Article の JSON-LD 生成部分）**:

```ts
// generate.ts または render 用の partial 内で
const articleSchema = {
  "@context": "https://schema.org",
  "@type": "Article",
  "headline": geminiOutput.meta?.title ?? page.default_title,
  "description": geminiOutput.meta?.description ?? "",
  "datePublished": pageDates?.datePublished ?? buildDate,
  "dateModified": pageDates?.dateModified ?? buildDate,
  "author": { "@type": "Organization", "name": "RISEby inc.", "url": "https://riseby.net" },
  "publisher": { "@id": "https://aimoaas.com/#organization" }
};
```

**表示の補足**: 本文フッターに「最終更新: YYYY-MM-DD」を表示すると、人的メンテのシグナルになる（任意）。

### 4.2 BreadcrumbList の付与（PSEO）【実装済み】

**目的**: 検索結果のパンくず・サイト構造の明確化。

**実装内容**: `templates/pseo/page.html` の `<head>` に `{{json_ld_breadcrumb}}` を追加。`generate.ts` で「ホーム → リソース → 当ページ」の BreadcrumbList を生成。既存 HTML は generate 再実行で更新。

**具体案（リソースハブ作成後）**: 2番目の item を `/ja/resources/` に変更可能。

---

### 4.3 著者・更新日の表示（オプション）【実装済み】

**目的**: 人的関与を「見える化」する。

**実装内容**: テンプレの TL;DR 直下に「提供: RISEby inc. / AIMOaaS™　最終更新: {{date_modified_display}}」を追加。`date_modified_display` はビルド日を「YYYY年M月D日」で表示。generate 再実行で反映。

**具体案（ページ別日付を導入する場合）**:

```html
<p class="text-sm text-slate-500 mt-2">提供: RISEby inc. / AIMOaaS™　最終更新: {{date_modified_display}}</p>
```

`date_modified_display` は `dateModified` を「YYYY年M月D日」などで表示する用。未設定なら「—」または非表示。

---

### 4.4 OGP 画像の差し替え（ページ別）

**目的**: 全ページ同一画像による「テンプレ量産」シグナルを弱める。

**具体案**:
- トピック別に 3〜5 種類の OGP 画像を用意（例: Evidence Pack 系 / ワークフロー系 / 責任分界系）。
- `catalog.yaml` の `page_type` や `topic` に応じて `og_image` を切り替える。
- テンプレの `{{og_image}}` を、ページごとの変数で上書きする。

画像が無い場合は、現状のデフォルト画像のままでも可。優先度は中。

### 4.5 リソースハブの追加（/ja/resources/）

**目的**: リソースと PSEO を一つの「テーマ」としてまとめ、トップからリンクしやすくする。

**具体案**:
- `ja/resources/index.html` を新設。
- 内容: タイトル「リソース」、短文説明、以下のブロック:
  - 「ガイド・用語」: ai-governance-guide, glossary, shadow-ai, eu-ai-act, human-in-the-loop, governance-as-code, maturity-checklist, case-studies, shadow-ai-governance-guide, aimo-analysis-engine へのリンク。
  - 「監査・証跡（実務）」: allowlist に入っている 20 の PSEO へのリンク（または代表 5〜10 本＋「監査・証跡の記事一覧」へのリンク）。
- このページを sitemap に追加し、`ja/index.html` の「リソース」リンクを `resources/ai-governance-guide/` から `resources/` に変更（または「リソース」でハブ、「AIガバナンスガイド」で個別にリンク）。

---

### 4.6 トップからの PSEO 代表リンク

**目的**: クロールとリンクエクイティの付与。

**具体案**: ja/index のフッターまたはリソースセクションに、「監査・証跡の実務」のような見出しで、allowlist の代表 3〜5 本へのリンクを追加（例: Evidence Pack ガイド、最小要件、申請・審査・例外、棚卸・継続監査）。同じリンクを多言語トップにも同じ構成で入れるとよい。

---

### 4.7 FAQ 数の充足と FAQ スキーマの拡大

**目的**: ユーザー利便性とリッチリザルトの機会を増やす。

**具体案**:
- 品質レポートで「faq_count < 8」のページは、Gemini の生成 FAQ を 8 問以上に増やすようプロンプトまたは後処理で対応。
- `faq_schema_allowlist.json` は、allowlist と同期させるか、index 許可ページは原則 FAQ スキーマも出してよい方針にし、スキーマの重複・不自然さだけ lint でチェックする。

---

### 4.8 内部リンクのトピック多様性

**目的**: 「page_types_in_links >= 3」を満たし、サイトのテーマの広がりを示す。

**具体案**: `link_map.json` または内部リンク生成ロジックで、同一ページタイプに偏らないよう、Evidence / Workflow / Responsibility 等のタイプをまたぐリンクを優先して組み込む。`scripts/pseo/postprocess_internal_links.ts` や link_map を生成する処理を修正。

---

### 4.9 noindex 方針の明文化と allowlist 拡大ルール

**目的**: 意図的な段階解放と、拡大時の品質維持。

**具体案**:
- `docs/` に「PSEO インデックス方針」を 1 ページでまとめる。
  - 基本: 未許可は noindex。sitemap には allowlist のみ。
  - allowlist 拡大: 品質レポートで pass かつ warn が少ない（例: faq_count >= 8, page_types_in_links >= 3, content_similarity が閾値以下）ページから人間が選んで追加。
- `index_allowlist.suggested.json` は release_candidates で出力済みなので、その候補をレビューしてから `index_allowlist.json` にコピーする運用を続ける。

---

### 4.10 Core Web Vitals（本番 CSS）

**目的**: LCP 等の改善。

**具体案**: 本番では Tailwind をビルドして 1 つの CSS にし、`<head>` でプリロード。Google Fonts は `display=swap` のままでも、フォント URL を preconnect 済みなら許容範囲。PSEO は静的 HTML のため、CDN キャッシュが効いていれば影響は限定的。優先度は中。

## 5. 優先度と実施順序の提案

| 優先度 | 項目 | 理由 |
|--------|------|------|
| 高 | Article に datePublished/dateModified/author/publisher | E-E-A-T と AI 低価値回避に直結。 |
| 高 | BreadcrumbList の付与 | 実装が軽く、検索結果の見え方が良くなる。 |
| 高 | リソースハブ（/ja/resources/）の新設とトップの「リソース」リンク先の見直し | サイト構造とクロールの要。 |
| 中 | 著者・更新日の表示（1 行） | 人的関与の見える化。 |
| 中 | トップからの PSEO 代表リンク 3〜5 本 | リンクエクイティと発見性。 |
| 中 | FAQ 数 8 以上への統一と FAQ スキーマの allowlist 拡大 | 品質レポートの warn 解消とリッチリザルト。 |
| 中 | 内部リンクのトピック多様性（page_types_in_links） | テーマの広がり。 |
| 低 | OGP 画像のトピック別差し替え | テンプレ臭の軽減。 |
| 低 | 本番用 Tailwind ビルド | CWV の余裕を持った改善。 |

## 6. まとめ

- **現状の「allowlist のみ index・sitemap 掲載」は、Scaled Content リスクを抑えつつ SEO を試すうえで妥当。**
- 問題の多くは「インデックス対象ページの価値の見え方」と「サイト構造」に集約される。Article の日付・著者・publisher、BreadcrumbList、リソースハブ、トップからのリンクで、検索エンジンとユーザーに「人的に関与した有用なリソース」であることを示すことが有効。
- 上記の具体策を優先度に従って実装し、allowlist の拡大は品質レポートと人間のレビューをセットにすることで、2026年2月時点のファクトベースの基準に沿った効果的な SEO に近づけられる。

## 7. 実施メモ（allowlist 20 継続・PSEO 分離方針）

- **allowlist**: 20 ページに意図的に絞り込んだまま継続。変更なし。
- **PSEO と一般リソースの分離**: PSEO は専門家向けとして、現在のリソース（ai-governance-guide 等）とは切り離して扱う。PSEO ディレクトリ内から他ページ（AIMO Standard、お問い合わせ等）へのリンクはそのまま許可。
- **実装済み**:
  - **OGP 画像**: `page_type` に応じる関数を追加（現状は同一画像。トピック別画像追加時に差し替え可能）。
  - **Breadcrumb**: PSEO 記事は「ホーム → 監査・証跡（実務）→ 当ページ」。2番目は `/ja/resources/pseo/` へリンク。
  - **テンプレヘッダー**: PSEO 記事のパンくずを「リソース」ではなく「監査・証跡（実務）」に変更し、`/ja/resources/pseo/` へリンク。
  - **PSEO ディレクトリトップ**: `/ja/resources/pseo/index.html` を新規作成。`scripts/pseo/render_pseo_index.ts` で `pseo_pages.json` から生成。トピック別セクション（証拠パック、監査・統制、申請・審査・例外、棚卸、責任分界、Coverage Map）で一覧表示。デザインは既存 PSEO と統一（Noto Sans JP、slate/indigo、実務的な UI）。sitemap に本 URL を追加。
  - **FAQ 数**: Gemini 出力の検証で 8 問未満の場合はエラーに変更。
  - **内部リンク多様性**: `buildInternalLinksHtmlFromLinkMap` で、自トピックに加え他トピックから 2 種を追加し、`page_types_in_links` の改善を図る。
- **一般リソースハブ（/ja/resources/）**: PSEO 分離方針のため、現状は新設していない。トップの「リソース」リンクは従来どおり。
